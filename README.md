# sotusei_readme
## 1. サービス概要

- このサービスはどんなサービスか（3行以内）
推し活で見つけた開始日や締切日のある情報をURLごと保存し、忘れないように管理できるアプリです。
見つけた情報をタイトル・日時・メモと一緒に記録し、通知によって行動のタイミングを逃さないようにします。

- 誰の、どんな課題を、どう解決するか
推し活でチケット先行やグッズ販売の情報を追っている人が、開始日や締切日を忘れてしまう課題を、URL保存・日時管理・通知機能によって解決するです。
---

## 2. このアイデアはどこから生まれたか

### 2-1. きっかけとなった体験・感情
私自身が推し活をしている中で公式のXやLINE、サイトから情報を収集することが多くチケットやグッズの情報も収集していました。
その場では「あとで申し込もう」「あとで買おう」と思っていて申し込みや購入をせず後回しにすることがありました。
時間が経ち、Xのポストで購入日時が過ぎていたことがあり受注販売だったこともありとてもショックを受けました。
カレンダーを使おうとも思いますがカレンダーを使うのが苦手なことと予定を入れた際に日付けを間違えて入れてしまうこともよくあります。
また発売開始の日時を忘れてしまい数量限定のものを購入できず「発売を知っていたのに…」「情報を知っていたのに…」と悔しい思いをしました。

### 2-2. なぜそれが「気になった」のか
推し活をしていく中で推しの情報はとても重要なものだと思っています。
情報だけでなく「知っていたのに忘れて変えなかった」こんな悔しいことはありません。
人によって推し活への思いはそれぞれですが後悔のない推し活をしたいという思いが強く「忘れてしまう」ということにもったいなさを感じました。
また忘れっぽいというところから一覧で見れたらわかりやすいなと感じました。
またカレンダーの手入力でミスを経験したこともありアプリで解決できないかというのも考えました。
また私自身、多ジャンルで好きなものが多いためどのジャンルでも使いやすく購入開始や締め切りなどのタイミングがわかったらいいなと思いました。


## 3. 課題の整理（表面的になっていないか）

### 3-1. 表に見えている困りごと

推し活で見つけた開始日時や締め切り日時など後回しにすることで重要なことを忘れてしまったり手入力によって日付けを間違えたりしてしまうことです。

---

### 3-2. 本当に解決したい課題は何か

・見つけた情報を後回しにしてしまいそのまま忘れてしまう。
・カレンダーに入れても日付けを間違えてしまう。
・推し活では、一度タイミングを逃すと受注や数量限定など取り返しづらいものが多い。

→ 本当に解決したい課題：推し活で見つけた大事な情報を、手間なく残し、タイミングを逃さず行動につなげられないこと。

### 10-2. 技術選定の理由

今回チャレンジしたい点

①Hotwire (Turbo/Stimulus) によるUXの追求
ページ全体をリロードせずに、保存完了や通知設定の切り替えを瞬時に行うフロントエンド体験の実装。

②外部メタデータの動的取得 (MetaInspector)
URLを入力した際、裏側でそのサイトのタイトルや画像を自動解析してフォームに補完する機能。ユーザーの入力負荷を極限まで減らす試みです。

③正確なスケジュール通知 (Sidekiq / Cron)
大量のデータの中から「通知すべき時間」を正確に判定し、非同期処理によってサーバーに負荷をかけずに確実にリマインドを送信する仕組みの構築。

④LINE Messaging API との連携(ここに関しては料金もかかるため検討中)
外部APIと連携し、Webサービスを飛び越えてユーザーが最も日常的に利用するLINEへ情報を届ける体験の実現。

不安な点と対策

① 外部サイトの仕様変更によるデータ取得の失敗
不安：外部サイトの構造や仕様変更により、タイトルなどの情報が正しく取得できない可能性がある。
なぜ問題か：
自動取得に依存しすぎると、情報登録時にエラーが発生し、ユーザーの操作が止まる恐れがある。
対策：自動取得に失敗した場合でも、タイトルや日時を手動で入力・編集できるUIを用意し、必ず登録できるようフォールバックを設計する。
これにより、機能が一部失敗しても、ユーザー体験が途切れないようにする。

② 通知処理の遅延や失敗
不安：サーバー負荷や外部サービスの障害により、通知が遅延・失敗する可能性がある。
なぜ問題か：本アプリは「タイミングを逃さないこと」が価値のため、通知の信頼性が低いとサービス価値が損なわれる。
対策：非同期ジョブにおいて再試行（Retry）設定を行い、一時的な失敗からの復旧を可能にする。
また、ログによる監視・追跡を行い、問題発生時に原因を特定できるようにする。
将来的にはRedisなどのインメモリDBを活用し、高負荷時でも安定した通知処理が行える構成を検討する。
